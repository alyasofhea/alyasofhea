Practical Work

Lab Exercises

LAB 01

UI Fundamentals

Building the first Jetpack Compose screens — exploring layouts, colours, and the Compose declarative paradigm from scratch.

This lab taught me that UI in Compose is data-driven, not view-based. Struggling with Column/Row alignment actually deepened my understanding of how the system measures children. I left with a solid mental model I kept reaching for in every lab after.

LAB 02

User Interaction

Adding state, click handlers, and basic animations to make screens feel alive and responsive to user input.

Managing state correctly was the hardest part — I kept triggering unnecessary recompositions until I understood remember and mutableStateOf properly. Once it clicked, building interactive UI felt genuinely fun. This lab changed how I think about reactivity.

LAB 03

Material Design 3

Applying Material You theming — dynamic colour schemes, typography scales, and component variants for a cohesive visual system.

Material Design 3 is far more flexible than I expected — the colour-role system (primary, secondary, tertiary) means your whole app adapts with one palette change. Applying it to Sanvello's calming green theme felt natural and intentional rather than arbitrary. I now design with roles, not raw hex values.

LAB 04

Navigation Component

Wiring up multi-screen flows using Jetpack Navigation — back-stack management, argument passing, and deep links.

Navigation graphs made me think about app architecture differently — screens are destinations, not activities. Getting argument passing right between screens was tricky, but once I did, the whole app felt coherent. This lab directly shaped how I structured both projects.

LAB 05

Room Local Database

Persisting data locally with Room — defining entities, DAOs, and observing Flow-based queries from the ViewModel layer.

Room's annotation-based approach felt magical until I understood what it generates under the hood. Connecting a Flow query to a Compose UI so the list re-renders automatically on data change was a genuinely satisfying moment. This lab completed the MVVM picture for me end-to-end.

Assessed Work

Projects

PROJECT 01 · Lab 4 + Working Features

Sanvello — Mental Wellness Companion

Problem Statement Many students struggle with daily stress and emotional burnout without a private, accessible outlet. Sanvello provides a calm digital space to log moods, set daily intentions, and read uplifting quotes — entirely on-device.

Multi-screen navigation, mood logging, daily goal setting, and a profile screen — all wired through Jetpack Navigation with a Material 3 green wellness theme.

Building a full navigable app for the first time revealed how important architecture decisions made early on either help or haunt you later. I learnt to design my NavGraph before writing a single screen. Seeing a real app I could show friends made every late night worth it.

PROJECT 02 · Lab 5 + API + Sensors + Cloud

Sanvello 2.0 — Full-Stack Wellness App

Problem Statement Mental wellness tools often feel disconnected from real daily behaviour. Sanvello 2.0 bridges that gap by combining local mood tracking (Room), real-time motivational quotes (ZenQuotes API), step counting (hardware pedometer sensor), and a shared community board (Firebase Firestore) — all in one app.

Extends Project 1 with Room persistence, a ZenQuotes REST API integration via Retrofit, a hardware step-counter sensor, and Firebase Firestore for a live community board.

Integrating four distinct data sources — local DB, a REST API, a hardware sensor, and a cloud database — in a single ViewModel layer taught me how important separation of concerns truly is. Firebase's real-time listener felt like a superpower once it updated the UI automatically. This project is the most technically complex thing I have ever built.

Learning Journey

Looking Back

When I started this module I thought mobile development meant drawing pretty screens. By the end I understood it means designing data flows, managing state across layers, and making architecture decisions that ripple through every line of code you write. Each lab built deliberately on the last — Compose layout, then state, then theming, then navigation, then persistence — and that scaffolding made the projects feel achievable rather than overwhelming. The hardest moments, a ClassCastException at runtime, a sensor that never fired, a Firestore rule that blocked every write, turned out to be the most educational ones because they forced me to actually read documentation rather than copy examples. I leave this module not just able to build Android apps, but able to read an error, form a hypothesis, and debug systematically — and that skill will outlast any specific API.
